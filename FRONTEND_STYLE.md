# 前端风格规范

本文件是本项目的前端视觉与交互设计基准。任何新建、重构或优化前端页面、组件、样式、动画、交互状态时，都**必须先遵循本文件**，再结合具体业务需求实现。

## 适用技术栈

- 前端：Vite + React + TypeScript
- 样式系统：Tailwind CSS + CSS variables
- 组件基础：shadcn/ui
- 图标：lucide-react
- 动效：优先 CSS transition / transform / opacity；复杂动效才引入 framer-motion
- 后端：FastAPI，前端通过 API 调用后端服务
- 部署：前端构建为静态文件，由 Nginx 或 FastAPI 静态服务承载

## 总体目标

做出来的界面必须像一个可演示的真实产品，而不是临时测试页。

项目默认方向：

**深色简洁科技风：深色背景、高对比文字、干净的卡片层级、克制的强调色、清晰的功能分区。视觉上追求"专业工具感"而非"科幻大片感"。**

## 核心设计原则

1. 先定风格，再写代码。
  - 每次设计页面前，先明确风格方向、目标用户、核心场景和一个记忆点。
  - 不允许只按“常规后台页面”“普通聊天页”直接生成。
2. 避免通用 AI 生成感。
  - 不使用默认白底灰卡片 + 蓝/紫按钮的普通组合。
  - 不使用泛滥的紫蓝渐变作为默认高级感。
  - 不使用完全居中的普通 hero 作为默认布局。
  - 不依赖 Inter / Roboto / Arial / system-ui 作为唯一视觉个性。
3. 保持强产品感。
  - 页面要有清晰主操作。
  - 聊天、语音、状态、错误、加载都要有可见反馈。
  - 视觉装饰必须服务于功能，不做无意义堆砌。
4. 约束优先于堆料。
  - 每个页面最多使用 1 个主色、1 个中性色体系、1 个强调色。
  - 间距使用 4px 或 8px 基线。
  - 圆角、阴影、边框、模糊、动效时长要形成统一系统。

## 推荐视觉方向

默认采用“未来感语音聊天控制台”：

- 背景：纯色深色基底（如 `#0a0e17`），不使用渐变、网格、噪点或光晕。
- 主色：仅 1 个强调色（如青蓝 `#38bdf8`），用于按钮、链接、状态指示，不用于大面积装饰。
- 字体：正文使用系统默认无衬线字体即可，不强制引入特殊字体。
- 布局：功能分区清晰，留白合理，聊天区为主，辅助信息靠边或折叠。
- 装饰克制：不使用发光阴影、流动光效、动态网格等装饰元素。视觉层级通过边框、背景色深浅、字号差异建立。

可选替代方向：

- 温暖陪伴式助手：浅暖背景、柔和卡片、低饱和强调色，适合情绪陪伴或教育类题目。
- 极简科技控制台：大量留白、细线分隔、单色强调，适合工具型或数据型题目。
- 深色简洁界面：深色背景、高对比文字、干净卡片、无发光装饰，适合通用型题目。

## React 与组件实现要求

- 优先使用 shadcn/ui 组件，不重复造 Button、Input、Dialog、Sheet、Card、Tabs 等基础组件。
- 使用 TypeScript，组件 props 必须有明确类型。
- 组件可以复用，但不要为了复用过早抽象。
- 图标统一使用 lucide-react，不使用 emoji 作为 UI 图标。
- 不引入额外 UI 库，除非明确说明收益和必要性。
- 业务组件命名要表达业务含义，例如 `ChatPanel`、`VoiceControl`、`ModelStatus`、`MessageBubble`。
- 样式优先通过 Tailwind utility 和 CSS variables 表达，避免散乱硬编码颜色。

## Tailwind 与主题要求

- 使用 CSS variables 管理主题 token：背景、前景、主色、强调色、边框、muted、ring、radius。
- 使用语义颜色，而不是在组件里到处写孤立色值。
- 同一个页面不要混用 warm gray、cool gray、slate、zinc 等多套中性色。
- 暗色模式不是简单反色，应重新校准背景层级、边框透明度和发光强度。
- 如果使用玻璃效果，必须控制透明度、边框和内阴影，避免廉价毛玻璃。

## 交互状态要求

所有可交互元素都必须覆盖：

- default
- hover
- active / pressed
- focus-visible
- disabled

聊天和语音相关界面必须覆盖：

- loading：模型正在响应、语音正在处理、消息正在发送
- error：API 错误、网络错误、语音合成失败
- empty：没有聊天记录或没有可播放语音
- streaming：模型流式输出时的进行中状态
- success：操作完成后的轻量反馈

## 动效要求

- 默认使用 CSS transition。
- 动画优先使用 `transform` 和 `opacity`。
- 不动画 `top`、`left`、`width`、`height`。
- 页面首次进入可以有轻量淡入（opacity 0→1，200ms）。
- 按钮点击可以使用轻微 `scale(0.98)` 或 `translateY(1px)`。
- 流式输出可以使用光标闪烁动效，但除此之外不使用持续微动效。
- 必须尊重 `prefers-reduced-motion`。

## 响应式要求

- 移动端优先，375px 宽度下必须可用。
- 桌面端可以使用左右分栏、侧边状态栏或控制台布局。
- 移动端必须退化为单列布局，不允许横向滚动。
- 聊天输入框、发送按钮、语音按钮在移动端必须容易点击。

## 可访问性要求

- 表单控件必须有 label 或明确的 aria-label。
- 图标按钮必须有可访问名称。
- focus-visible 状态必须清楚可见。
- 文本和背景对比度必须足够。
- loading 和 error 状态必须能被屏幕阅读器理解。

## 输出质量要求

- 代码必须完整可运行。
- 不留下 TODO、占位逻辑或未接线按钮。
- 不硬编码 API Key 或敏感信息。
- 不为了视觉效果破坏现有 FastAPI API 调用方式。
- 修改已有页面时，先理解现有结构，再最小范围改造。
- 新增依赖前必须确认项目是否已有对应依赖，并说明为什么需要。

## 每次前端实现前的检查清单

- 是否明确了本次页面的风格方向？
- 是否符合默认“未来感语音聊天控制台”或明确的新方向？
- 是否避免了普通 AI 风格默认值？
- 是否使用了统一主题 token？
- 是否覆盖 loading、error、empty、disabled、focus-visible？
- 是否在 375px 移动端可用？
- 是否适合 Demo 视频展示？
